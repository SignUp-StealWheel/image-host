三个范式

  • 1NF关系型数据库基本要求
  • 2NF非主键列（直接｜传递）完全依赖于主键列
  • 3NF非主键列直接完全依赖于主键列

1NF：

  • 每个字段都不可拆分

反例：

字段 userInfo: 广东省 10086' ，依照第一范式必须拆分成 userInfo: 广东省 userTel:10086两个字段

2NF：

  • 表必须有主键
  • 非主键列必须完全依赖于主键列，不能部分依赖于主键

反例：

选课关系表

student_course(student_no, student_name, age, course_name, grade, credit)

主键为(student_no, course_name)

student_name部分依赖于主键student_no

grade部分依赖于主键course_name

存在的问题

  • 数据冗余：一个学生选多门课程会冗余多份student_name、age信息
  • 插入异常：插入一门课程时，此时没有绑定学生会导致缺少学号而无法插入

可以拆分成三个表：学生：student(stuent_no, student_name, 年龄)；课程：course(course_name, credit)；选课关系：student_course_relation(student_no, course_name, grade)

3NF：

  • 非主键列必须直接依赖于主键，不能存在传递依赖

学生关系表为Student(student_no, student_name, age, academy_id, academy_telephone)

主键为"学号"，其中学院id依赖于学号，而学院地点和学院电话依赖于学院id，存在传递依赖，不符合第三范式。

存在的问题：

  • 数据冗余：同一学院多个学生冗余了多份academy_id, academy_telephone数据
  • 插入异常：新创建一个学院时，此时缺少学生信息会导致无法插入

可以把学生关系表分为如下两个表：学生：(student_no, student_name, age, academy_id)；学院：(academy_id, academy_telephone)。

mysql中存储引擎分类

MyISAM、InnoDB、MEMORY、ARCHIVE

InnoDB：目前的默认引擎

优点：支持事务，支持崩溃恢复，基于聚簇索引，支持表锁、行锁，支持外键

缺点：占用空间大

MyISAM：

优点：访问速度快

缺点：不支持事务、不支持崩溃恢复、支持表锁、不支持行锁、不支持外键

Memory：

优点：访问速度快

缺点：内存引擎，崩溃后数据全丢，hash索引不支持排序、范围查询、部分索引匹配

Archieve：

优点：压缩占用空间小，插入快，适用于归档数据

缺点：不支持索引，查询慢

drop、delete与truncate的区别是什么

三者都表示删除，drop是删除一张表，delete是删除记录，truncate是重置表并清空记录。

drop和truncate属于DDL，一经操作不可回滚，而delete在未commit之前可以回滚。

各字段存储需求：

char特点

  • 定长，可指定0-255字符，实际占用空间由字符集而定，例如utf8一个字符3个字节则最大空间为255*3
  • 可以有默认值
  • 截断尾部空格

varchar特点

  • 非定常，溢出存储，只在索引记录中存储768字节，其余数据存储到overflow page溢出页中
  • 可以有默认值
  • 最大65535字节，实际可用字节需减去下述字节
  • 长度>255则需要占用2个字节存储长度
  • 长度<=255则需要占用1个字节存储长度
  • 如果允许为空则需要占用1个字节标明允许为空，不允许为空则不占字节

utfmb4编码的varchar列，允许为空，则最多可存储(65535-2-1)/4=16383字符

ascii编码的varchar列，允许为空，则最大可存储(65535-2-1)/1=65532字符

ascii编码的varchar列，不允许为空，则最大可存储(65535-2)/1=65533字符

TEXT特点

  • 长字符串型，定长
  • 不可以有默认值
  • 溢出存储，只在索引记录中存储768字节，其余数据存储到overflow page溢出页中

redo log、undo log、bin log

bin log是mysql server层逻辑日志，记录对数据库进行的所有变更操作，和引擎无关，主要用于恢复数据库和同步。对于innoDB，binlog对每个事务仅在commit之前记录一次。

redo log是innoDB引擎专用的物理日志结构，用于描述对底层数据页进行的具体操作，记录物理页的修改。无论事务是否提交，都会先将操作写入到redolog中，当mysql崩溃重启时，结合binlog进行数据恢复或回滚，保证数据持久性。

undo log是innoDB引擎专用的逻辑日志，实现原子性，用于事务回滚及mvcc实现。事务修改先记undo，undo修改先记redo。undo log在事务修改前保存了一份原始数据、保存了修改后的数据版本并持久化到了磁盘中后才最终执行了事务的commit。当InnoDB存储引擎回滚时，它实际上做的是与先前相反的工作。对于 每个INSERT，InnoDB存储引擎会完成一个DELETE;对于每个 DELETE，InnoDB存储引擎会执行一个INSERT;对于每个UPDATE， InnoDB存储引擎会执行一个相反的UPDATE，将修改前的行放回去。mvcc并不是存储了所有的版本，而是根据当前版本+undo log，重新计算出来所需版本的。

什么是MySQL的GTID

Global Transaction ID，全局事务id，由SOURCE_ID：TID组成，其中sourceid是mysql实例的唯一表示，TID是事务提交时才分配的值，单调递增。基于GTID搭建主从时，可保证每个事务只会执行一次。

什么是分区表

分区表是一个独立的逻辑表，但是底层是由多个物理子表构成。查询数据时只会去具体的子分区中查询。但业务中禁用分区表，因为：

  • 分区表加锁时会锁住全部分区，而不是单独的分区

分区表类型

list、hash、range

分库分表和表分区的区别

表分区是单机实例下的操作，所有的数据都在一个服务器上，对外相当于一张逻辑表。

分库分表是将一张表分解成多张表，数据可分布在多台实例上，需要代码协助实现。

分库分表标准及类型

标准：

实际业务数据量达不到分库分表阈值，因此对这一块只是了解概念，但并无实操经验。

类型：

垂直分库：将不同的表放到不同的数据库

水平分库：将同样的表结构放到不同的数据库上，并根据规则将数据分发

垂直分表：将不同的列放到不同的表中

水平分表：将同样的表结构放到不同的表上，并根据规则将数据分发

主键使用自增ID还是UUID

推荐使用自增ID，在innoDB引擎中，主键索引是作为聚簇索引使用的，也就是B+树中按主键排序后存储了所有的数据。如果是自增ID，新增时只需往后追加即可，但UUID是无序的，插入时可能导致页分裂，造成非常多的数据的移动，并产生内存碎片。

delete大量数据后，为什么表空间大小不变

因为delete操作只是标明B+树中某个位置可复用，并不会删除这个位置。

服务器Binlog日志导致磁盘满，怎么办

  • 设置binlog过期时间

    • 5.7默认永久有效，set global expire_logs_days=10配置10天过期
    • 8.0默认30天过期，set global binlog_expire_logs_seconds=86400配置指定时间过期

  • 手动删除Binlog日志

    • show master status 查看当前正在使用的日志
    • show binary logs 查看所有日志
    • purge binary logs to 'mysqlhost01-bin.000010'，删除0.000010之前的所有binlog日志，不包括0.000010
    • PURGE MASTER LOGS BEFORE '2003-04-02 22:46:26'，删除置顶日期之前的日志

注意事项：

如果有主从同步，一定要先查看从库同步到哪个binlog了，如果删除了从库未同步到的binlog，会导致数据缺失。

索引

索引有什么分类

普通索引：普通的索引

唯一索引：普通索引+唯一限制

主键索引：特殊的索引，不允许有空值

MySQL数据库中索引的工作机制是什么

索引用于快速定位，是一个排序的数据结构。一般使用B+树进行实现。

唯一和主键索引的区别

一张表可以存在多个唯一索引，值可为null，但一张表最多存在一个主键索引，值不可为null

聚簇索引和非聚簇索引

聚簇索引的叶子结点是数据，非聚簇索引的叶子结点依旧是索引

innoDB中主键索引默认为聚簇索引，如果没有主键则唯一非空索引替代，如果不存在则隐式定义一个主键作为聚簇索引。因为隐式主键在分配到最大值后，会归0并覆盖原有数据，因此非常建议每个表都明确指明一个主键。

InnoDB存储引擎有哪些常见的索引

B+树：常用结构

Hash：把索引值做hash运算，特点如下

xxxxxxxxxx

说一下B+树与B树的区别

B树是一棵平衡多路查找树，所有叶子结点都在同一层并且每层都有数据。

B+树是B树的变种，非叶子结点仅保存索引，所有数据都保存在叶子结点中，并且叶子结点之间指针相连，按顺序排列，大大增加了区间访问的效率。

B+树的查找时间固定为logn，B树的查找时间不固定最快o(1)，最慢log(n)

B+树由于非叶子结点不存储数据，所以同样的一页空间，B+树可以保存更多的结点，因此相比B树高度更低。

为什么B+树比B树更适合实现数据库索引

  • 非叶子结点仅存放索引，一页可放更多的数据，树的高度低，因此查询时IO次数少
  • 叶子结点之间指针相连按序排列，更适合区间查询
  • 数据仅存放在叶子结点，所有的查询效率一致为logn，比较稳定

Hash 索引和 B+ 树索引区别是什么？你在设计索引是怎么抉择的

  • B+支持范围查询，Hash不支持
  • B+自带排序，Hash不支持
  • B+支持最左匹配，Hash不支持
  • B+使用Like查询时，如果通配符在后面依旧可以使用索引，Hash不行
  • Hash等值查询速度比B+快

如何选择在哪些列上建索引

  • 主键索引每张表必建一个
  • 区分度高的列建索引，像性别这种列建索引就不合适
  • 经常当作条件查询的列建索引
  • 作为多表关联字段的列适合建索引
  • 变化较少的列上建索引，变化较多的列会导致索引经常变化

什么是最左匹配原则

最左匹配原则和联合索引有关，例如a,b,c三个字段的联合索引，a全程有序，b在a相等的小范围内有序，c在a和b都相等的小范围内有序。因为是按照从左往右顺序匹配，直到遇见范围查询(> < like in between等)则后续字段不走索引，如果不匹配a，就无法使用b，因此叫最左匹配原则。

什么是覆盖索引

select 查询的列在索引中，因此无需回表查询即可直接从索引中取出来数据返回，叫做覆盖索引。但并不是每种索引类型都支持覆盖索引，像Hash索引由于不在索引结构里存储实际数据，因此不支持覆盖索引。

索引的设计原则

  • 数据区分度高
  • 索引尽量短（索引越短，一页索引越多，IO次数越少）
  • 不是越多越好，索引越多，增删改时需要维护的时间越多
  • 尽量选择变化少的列建立索引
  • 尽量应用最左前缀原则

索引什么时候会失效

  • 组合索引不符合最左匹配原则
  • 以%开头的like
  • 列是字符串，但是查询条件没加引号
  • 非主键列使用 != <> not in 等运算符
  • 使用or且任意条件未加索引
  • 对索引列进行运算
  • is not null会导致索引失效
  • 左右连接时数据类型或编码格式不一致
  • mysql估计全表扫描比使用索引快的时候

注意：exists对外层查询是否使用索引没有影响，和网上说法不一致

什么是前缀索引

有时会对较长的字符串添加索引，这时会导致索引比较大，影响效率。前缀索引可以对字符串的指定前多少个字符创建索引，这样索引更小，效率更高。

MVCC实现原理

multi-version concurrency control，多版本并发控制。mysql使用mvcc实现了快照读，并提供回滚保证了原子性。具体实现是同一数据多个版本，并通过事务id来判断某事务对某版本是否可见。当数据更新、删除、插入时，不管事务有没有提交都会产生一个新的版本，并记录是哪个事务进行的操作。当一个新的事务开启后，会记录两组数据，1是开启瞬间当前未提交的事务id数组，2是未提交事务数组的事务id最小值和当前系统已分配最大事务id。当事务访问数据时，通过这两组数据可判断：

  • 小于最小值，必定已提交，因此可见
  • 大于最大值，必定未开启，因此不可见
  • 处于区间中，则需要根据开启瞬间记录的事务id数组来判断
  • 如果处于开启瞬间记录的事务id数组中，则不可见
  • 如果不处于开启瞬间记录的事务id数组中，则不可见

快照读和当前读

快照读是基于mvcc实现的非加锁读取方式，普通的select语句就是快照读。

当前读是读取的最新版本，update、delete、insert、select...lock in share mode、select...for update等都是当前读

分析器进行词法分析及语法分析

优化器决定使用哪个索引、各表连接顺序等，产出执行计划

执行器拿到执行计划，首先判断是否有操作权限

执行器按计划和引擎进行交互，将结果集返回客户端

更新语句执行过程

分析器进行词法分析及语法分析

优化器决定使用哪个索引、各表连接顺序等，产出执行计划

执行器拿到执行计划，首先判断是否有操作权限

执行器按计划和引擎进行交互

执行器先拿到要更新的行信息，然后变更值后再调用引擎接口写入变更后的数据

引擎将数据写入内存并写入redo log，此时redolog处于prepare状态，通知执行器写入binlog

执行器写入binlog完毕后调用引擎commit api提交事务，更新完成

where用于表

having用于组，搭配group by使用

where在group by之前做筛选，having在group by之后做筛选

什么是MySQL主从同步

mysql可以配置主从服务器，通过binlog，将主服务器上的变更操作重放到从服务器上。复制操作是异步进行的，从服务器不需要一直连接着主服务器，可以分多次断断续续执行，并且可以配置复制指定数据库、指定表等

为什么要做主从同步

  • 数据备份，保证数据安全
  • 主服务器写入数据，从服务器分析数据，分担主服务器压力
  • 读写分离
  • 数据库版本测试
  • 主备切换

乐观锁和悲观锁是什么

悲观锁假定每次执行都存在冲突

乐观锁假定每次执行都不存在冲突

悲观锁在每次执行前提前加好锁，拿不到锁不让执行，例如数据库的锁机制就是悲观锁

乐观锁给表添加version字段，每次更新时判断version字段是否相等，如果不相等则不能更新

锁

什么是锁，锁的作用是什么

锁是并发访问资源的一种保障机制，保证了数据的一致性和完整性。

InnoDB存储引擎中的锁都有哪些类型

共享锁 S锁：允许读

排他锁 X锁：允许更新和删除

间隙锁：gap lock

Next-Key lock：gap lock+行锁

意向共享锁 IS锁：表级锁，事务想要给表中某几行加锁

意向排他锁 IX锁：表级锁，事务想要给表中某几行加锁 锁的兼容性问题： 由图可知，当持有gap锁时，禁止添加insert intention。但持有insert intention时，是可以添加gap的，只是此时的gap被insert intention截断了，范围变小了。

锁可能会带来什么问题

脏读、不可重复读、幻读、死锁等。

数据库中的死锁概念你知道吗

死锁是因资源争夺而互相等待的现象。出现死锁必须满足4个条件：

  1. 资源互斥（同时只能被一个对象占有）
  2. 保持并请求（占有一个资源，并请求其他资源）
  3. 不可剥夺（自己不释放，就不会释放）
  4. 循环依赖（互相等待）

打破任何一个条件都会解除死锁，mysql提供了超时回滚和等待图死锁检测机制。

发现死锁应该如何解决？

  • show engine innodb status;查看最近一次死锁现场信息
  • 根据死锁日志找出死锁触发的sql语句、持有的锁、要请求的锁
  • 分析sql语句，优化加锁逻辑

事务

什么是事务

事务是数据库的一个原子操作单元，同一事务中的多条操作，要么同时成功，要么同时失败。

事务四大特性

A（atomicity）：原子性

通过redo log实现，事务操作要么全部成功、要么全部失败

C（consistency）：一致性

通过undo log实现，保证事务执行、回滚前后，数据依旧满足数据库设定的状态，包括约束、触发器、级联及三者的任意组合。

这里并不完全保证符合程序员的业务期望，这应该是代码层需要保证的事情，此一致性仅保证任何编程错误都不能导致违反任何定义的数据库约束。

Consistency (database systems)

Consistency (or Correctness) in database systems refers to the requirement that any given database transaction must change affected data only in allowed ways. Any data written to the database must be valid according to all defined rules, including constraints, cascades, triggers, and any combination thereof. This does not guarantee correctness of the transaction in all ways the application programmer might have wanted (that is the responsibility of application-level code) but merely that any programming errors cannot result in the violation of any defined database constraints.[1] Consistency can also be understood as after a successful write, update or delete of a Record, any read request immediately receives the latest value of the Record

I（isolation）：隔离性

通过锁实现，当前事务对其他事务操作的可见性，根据隔离级别不同可见性不同

D（durability）：持久性

通过redolog实现，事务一旦提交，其结果就是永久性的，即使数据库本身宕机，也能恢复。但外部因素例如服务器磁盘坏了导致的数据丢失，不在此持久性讨论范围内，这是HA（Hign Availability）的职责。

事务隔离级别有哪些

有四种隔离级别，分别是：

Serializable：序列化

所有事务线性进行，效率很低

RR（repeatable read）：可重复读

默认级别，

RC（read committed）：读已提交

RUC（read uncommitted）：读未提交

四种隔离级别分别能提供不同强度的隔离强度，序列化没什么可说的，线性执行效率最低。RR级别存在幻读问题，但innoDB使用mvcc技术，引入了next-key lock，解决了当前读的幻读问题，但快照读依旧存在幻读问题。

RR级别幻读示例1：

两个事务，A先执行一次范围读取，然后B在此范围内插入一条数据并提交事务，A对事务B新插入的记录进行更新，A再次执行相同的范围查询，会发现A将刚才更新的B的记录查出来了。

RR级别幻读示例2：

(id,a,b)

(1,1,1)

两个事务，A先查询id为1的记录，然后B将id为1的记录的b字段更新为2并提交事务，A将id为1的记录的a字段更新为3，此时A再次查询id为1的记录得到的是(1,3,2)，发现读出来了事务B对字段b的变更。

上述事例2属于幻读的范围而不是不可重复读，因为如果A不执行更新操作，select查询出的数据依旧是(1,1,1)

RC级别存在不可重复读和幻读两个问题，因为RC级别能读取到其他事务已提交的数据。以RR级别示例2为例，如果如果A不执行更新操作，select再次查询出的数据变化为(1,1,2)，这种现象叫做不可重复读。

RUC级别存在脏读、不可重复读、幻读三种问题。读取到其他事务未提交的数据叫脏读。

InnoDB为什么选用RR为默认的隔离级别？

个人认为在不考虑效率的情况下，serializable级别肯定是最安全的，但是我们肯定是要考虑效率的，RR级别相比于serializable会出现幻读，但RR提供了next-key lock，如果操作前正确的加上了next key lock，那幻读是可以避免的。因此默认选用了RR级别
